# 🏛️ BLUEPRINT: Task 5.1 - World-Class Feature-Based Architecture
> **Objective**: Transition ENGLABS from a flat monolith to a modular, feature-oriented structure based on industry best practices for high-scale React apps.

## 📁 TARGET DIRECTORY STRUCTURE
We will reorganize the codebase into the following hierarchy:

### 1. `src/features/` (Modular Domains)
Each feature will be self-contained:
- `features/inventory/`: Components, Hooks, API logic specifically for stock.
- `features/staff/`: Attendance, Payroll, and HRM logic.
- `features/projects/`: Fabrication stages and project costing.

### 2. `src/components/` (The Atomic Library)
Generic UI components that don't know about "Inventory" or "Staff":
- `atoms/`: `Button`, `Input`, `Badge`, `Spinner`.
- `molecules/`: `SearchField`, `FormField`, `ModalHeader`.
- `organisms/`: `NavigationSidebar`, `GlobalHeader`, `DataTable`.

### 3. `src/lib/` (The Engine)
- `services/`: `firestore.service.ts`, `ai.service.ts`.
- `hooks/`: `useAuth.ts`, `useTheme.ts`, `useNetwork.ts`.
- `utils/`: Formatting, date handling, and math.

## 🛠️ REFACTORPHASE: THE "STAFF" TEST
We will begin the refactor with **Task 5.2 (StaffView)**. 
- **Goal**: Break the 800-line `StaffView.tsx` into 5-6 smaller components.
- **Verification**: Use the existing `sync.roles.test.tsx` to ensure ZERO functionality is lost during the move.

## 🧪 SUCCESS CRITERIA
1.  **Zero Cycles**: No circular dependencies between features.
2.  **No "God Files"**: No single file exceeds 300 lines of code.
3.  **Strict Isolation**: A change in `features/staff` should NEVER require a change in `features/inventory`.
